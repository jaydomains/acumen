## Findings by severity

### HIGH (3)

**H1 · B2-1 · Shuffled MCQ/matching options graded against the original index → silent wrong scores** — confidence: high · **TRIAGE: FIX-NOW (priority tag)**
`app/domain/attempts.py:525-546` (`_present_one`), `:1201-1229` (`_grade_mcq`/`_grade_matching`); permutation produced at `:468-474` (`option_permutation`), never inverted. With `randomise_option_order=True` (fixture default), presentation physically reorders MCQ `options` / matching `rights` but emits **no stable id or original-index tag**; grading compares the submitted *presented* index against the *original*-order `config.correct` (and identity mapping for matching). Unless the permutation is identity, correct selections score 0 and some wrong selections score 1. Independently corroborated by Lane B3's cross-lane note. Uncaught because `test_p4_grading.py` autosaves original-order indices, bypassing the presentation layer. Fix direction: invert the permutation at grade time (`original_index = perm[choice]`), or give presented options stable ids.

**H2 · F3-1 · MCQ "mark correct" radio wipes typed-but-uncommitted choice text** — confidence: high · **TRIAGE: FIX-NOW (priority tag)**
`src/app/(authed)/(admin)/admin/tests/[testId]/edit/_components/mcq-choices.tsx:46-52`. `setCorrect` loops `update(i, {...f})` over the `useFieldArray` `fields` snapshot, but choice text is an **uncontrolled** `register`ed input — the snapshot carries stale (empty) text, so writing it back discards what the admin typed, propagating to the POST body / "Choice text is required" zod error. Uncaught (existing test never marks a choice correct). Fix direction: read live values via `form.getValues("config.choices")` or `setValue` only the `.correct` field.

**H3 · B3-1 · `delete_test` FK-violates / orphans Responses of an attempted frozen test** — confidence: med · **TRIAGE: QUEUE**
`app/domain/tests.py:247-259`, reachable via `DELETE /v1/tests/{id}` (`app/routers/tests.py:153-162`). Only guards campaign-lock (`_guard_unlocked`) — not published/attempted state. It deletes the `Question` rows that frozen-test `Response.question_id` references under a **RESTRICT** FK (`models.py:620-624`, no `ondelete`) → unhandled 500 and destroys attempt history (contradicts AC-D17 snapshot retention). Postgres-only: the zero-DB FakeSession does not enforce FKs, so tests pass. Fix direction: block deletion of attempted/published tests, or soft-retire.

### MEDIUM (6) — all TRIAGE: QUEUE

**M1 · F2-1 · Autosave debounce fires (then retry-loops) against a paused attempt** — confidence: high · `also-A4`
`src/lib/attempts/use-attempt.ts:339-369`, `src/components/attempt/FrozenRunner.tsx:176-191`. `setAnswer` gates *scheduling* on `pauseState === "active"`, but an already-queued `setTimeout` still fires and `runAutosave` never re-checks pause state. The POST hits a paused attempt → server rejects → exponential retry loop (2s/4s/8s) → spurious persistent "save failed" banner. Fix direction: cancel pending debounce + retry timers on pause, or bail in the debounce callback when not active.

**M2 · B3-2 · Duplicate `pill_ids` on path create/update violate `uq_learning_path_pill`** — confidence: med
`app/domain/catalogue.py:307-327` (`_set_path_pills`), via `app/routers/paths.py:56-114`. No dedup in `LearningPathCreate/Update.pill_ids` (`schemas.py:247,253`) or the router's existence-only `_validate_pills`; two rows with the same `(learning_path_id, pill_id)` violate the unique constraint (`models.py:993`) → 500. Postgres-only (FakeSession ignores unique constraints). Fix direction: dedupe preserving first-seen order, or 422 on duplicates.

**M3 · B3-3 · `list_me_competence` derives `n` from `Test.pill_id` but profiles are keyed by `assignment.pill_id`** — confidence: med · `also-A3`
`app/domain/competence.py:626-657` vs `apply_competence_update` (`:485,515-527`). The two pill linkages are independent; when they diverge (common for generated/non-opted tests where `Test.pill_id` is NULL) the profile row gets `n=0`, so confidence is permanently "preliminary" and the AC-D20 confident threshold (default 20) never triggers even for testees with many submitted attempts. Fix direction: count `n` over the same linkage the profile uses (attempt → assignment → pill).

**M4 · B4-1 · `retrieve_for_generation` not fail-soft against embedding dimension mismatch** — confidence: med · `also-A3`
`app/domain/drive_rag.py:697-735`. Only the `provider.embed` call is wrapped in try/except; the `cosine_top_k` ranking at `:735` is not. After an embedding-model change leaves stale-dimension `DriveChunk` rows, `cosine_top_k` raises `ValueError` (`strict=True`) and aborts the entire generation/learning-material call instead of falling through to empty RAG context — contradicting the function docstring and SPEC §6.1. Fix direction: extend the try/except to cover ranking, return `[]`.

**M5 · B2-2 · Anchor pool top-up deficit miscounts when `assigned_difficulty != band`** — confidence: med
`app/domain/calibration.py:449-480` (`_live_anchor_counts_by_band`) vs `:602-609` (top-up loop). Deficit keys live anchors by `assigned_difficulty`, but the top-up loop reads `live_by_band.get(band, 0)` by loop `band`; `band_calibration_state` and `draw_anchors_for_attempt` correctly filter on `row.band`. When the AI-stamped `assigned_difficulty` diverges from `band`, the deficit is over-stated and the pool over-generated (extra AI cost). Fix direction: confirm the generation contract guarantees `assigned_difficulty == band`; if not, key the count on `row.band`.

**M6 · B3-4 · Direct `testee_ids` assignment targets are not role-filtered** — confidence: low · `also-A3`
`app/domain/assignments.py:135-157` (`create_assignment`). Direct targets are filtered only by active status; the system-group path enforces `ROLE_TESTEE` (`:84-87`). An admin can be made an assignee and then sees the test in `/v1/me/assignments`. Untested gap (no direct-target role-filter test). Fix direction: confirm intent (AC-D15/AC-D16); apply the role filter to direct targets if not intended.

## Cross-cutting patterns

1. **Presentation/grading shuffle contract is broken (H1).** Highest-impact defect, flagged independently by two lanes (B2 + B3). `randomise_option_order` shuffles the view but grading still expects original-order indices, with no key handed to the client. Affects MCQ *and* matching.
2. **The zero-DB FakeSession harness masks a class of data-integrity bugs (H3, M2).** FK RESTRICT and UNIQUE violations pass tests but fail on real Postgres. Dovetails with Audit 1's finding that the migration round-trip / `tests/e2e` could not run locally (no Postgres) — the DB-enforcement gap is a recurring blind spot. **Flagged as convergent for the Audit 5 synthesis.**
3. **Doc/comment ↔ implementation drift (M4, F1-1, F2-2).** Several "the docstring/comment promises X, the code does Y" cases — mostly `also-A3`; prime feed for Audit 3.
4. **Guards applied at call-time or create-time but not on the follow-on path (M1, M6, B1-2).** Pause gates scheduling but not queued timers; `TestUpdate` skips `TestCreate`'s matrix; token-consume skips the login-time deactivation gate.
5. **Inconsistent role/scope filtering (M6, B1-1).** Group resolution enforces role; the direct path does not.
