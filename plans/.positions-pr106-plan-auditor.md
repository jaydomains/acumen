## Positions

### S1-P1 — Input contract: topic→N, **not** name+desc→1 (the generator/refiner distinction)
A correct Slice 1 must specify the generator's **input** as a *topic prompt* (+ optional parent
subject, target count, difficulty envelope, seed reference material), explicitly **distinct** from
the built **refiner's** single admin-supplied `name` + `description`. The refiner is what exists
today: `enqueue_pill_proposal(... name, description ...)` (`app/domain/catalogue.py:488-516`),
`pill_proposal.py:27-32` ("Proposed name / Proposed description"). A Slice 1 that re-uses the
refiner's name+desc input — or fails to state the new topic→N contract — recreates the §2.2
"refiner, not a generator" gap. **Measured against:** `…workstream.md:135-145`,
`catalogue.py:488`, `pill_proposal.py:22-53`.

### S1-P2 — Output contract completeness (the §6.5 + §3 quality bar; folded A-1)
Each of the N drafts must carry **all** of: name, description, parent subject, **per-band
difficulty decomposition**, **grounding refs/citations**, **safety self-classification** (AC-D21),
rationale, **evidence count**, and an **explicit cited gap signal** ("must cite the specific gap
signal"; admin evaluates in 30 s). Dropping `evidence count` or the cited gap signal re-opens the
already-folded auditor finding **A-1**. **Measured against:** `SPEC.md:346,348`,
`DECISIONS.md:529,535` (AC-D21), `…workstream.md:138-145`, `…workstream.md:378` (A-1 fold).

### S1-P3 — G1 **surfaced, not baked** (extend `Operation.pill_proposal` vs mint new op)
Whether the generator **extends** `Operation.pill_proposal` or **mints** a new `Operation`
(e.g. `pill_generation`) is **ratification-class** (`REQUIRED_READING.md` §7(i)/(ii); workstream
**G1**). The refiner and generator have different input/output contracts, so this is a real fork.
A correct Slice 1 **holds G1 as a SURFACE** awaiting authenticated spec-author ratification and
does **not** silently pick a branch and build on it. **If the planner surfaces G1 → correct
SURFACE, not a finding.** A finding only arises if Slice 1 **bakes** a G1 choice (mints/extends as
settled) without ratification. **Measured against:** `…workstream.md:290`, per-item note
`:303-309`, `provider.py:121-157`, role files §7/§8.3.

### S1-P4 — G7 **surfaced, not baked** (prompt-registry version trajectory)
New prompt file + initial `VERSION`, and whether `pill_proposal` is **kept** or **replaced** (and
its version trajectory), is ratification-class (workstream **G7**, AC-CD8). Slice 1 must surface
G7 and not bake a version decision; the persisted-version contract (`catalogue.py:533`) means a
version choice has downstream provenance consequences. **Surface = correct, not a finding;** baking
= finding. **Measured against:** `pill_proposal.py:20`, `catalogue.py:530-537`, `…workstream.md:296`.

### S1-P5 — Shared-infra reuse, **no rebuild**
Slice 1 must **reuse** (not duplicate): the provider abstraction + `Operation`→`generate` routing
(`provider.py:121-157`), the versioned prompt registry with embedded `VERSION` persisted to
provenance (AC-CD8, `catalogue.py:530-537`), and `processing_tasks` async persistence (AC-CD7,
`app/models.py:1174`). N drafts persist as **N `processing_tasks` rows**; the existing
approve/reject pipeline (`catalogue.py:567-639`) + FE proposals tab
(`…/_components/proposals-tab.tsx`) are reused **unchanged**. A Slice 1 that proposes a parallel
persistence/queue surface is wrong. **Measured against:** `…workstream.md:115-126,160-162`.

### S1-P6 — AC-CD15 zero-network test discipline

# Pre-registered positions — Slice 2 (A2): grounding retrieval + per-draft citations (gated G4)

**Pre-registered:** 2026-06-07 (UTC), at planner Slice-1 convergence `18a719a`, **before the planner
posts Slice 2** (§4.2 anti-anchoring). Grounded against the live retrieval surfaces. Fresh finding-ID
set `S2-*`. Scope per parent §3.2 / §6 A2: *the generator retrieves reference context for the topic
(Drive RAG `retrieve_for_generation` and/or web search) and emits per-draft citations*; gated on **G4**.

### S2-P1 — Topic-keyed retrieval, NOT pill-keyed (the load-bearing mismatch)
The existing `retrieve_for_generation` (`app/domain/drive_rag.py:650`) is **pill-scoped** — signature
`(db, *, pill, target_difficulty, k)`, **returns `[]` when `pill is None`**, and builds its query from
the pill's name+description+band (`build_rag_query`). But pill **generation has no pill** — it grounds
a **topic**. A correct A2 plan must address this mismatch (generalize the query-builder to a topic
string, or add a topic-grounding variant) and must **not** naively call `retrieve_for_generation(pill=…)`
(which returns `[]` for a topic → silently un-grounded drafts). **Measured against:** `drive_rag.py:650-690`,
parent §3.2.

### S2-P2 — Reuse the established `rag_context` prompt pattern (AC-D22), don't reinvent
The built generation prompt already consumes a `rag_context` payload key (pre-rendered top-k Drive
chunks, P9/v1.1.0 AC-D22, `app/ai/prompts/generation.py:18-27`; empty renders `(none)`). A2's
`pill_generation` grounding should reuse this pattern (payload `rag_context` from retrieved chunks),
not invent a parallel mechanism. **Measured against:** `generation.py:18-27`, AC-D22.

### S2-P3 — `grounding_refs` is the NEW output-side field (per-draft citations)
The existing prompt *consumes* `rag_context` but does **not emit** citations. A2 adds the **output-side**
`grounding_refs` to each draft (the field Slice-1 §1.2(a) reserved + G7(7b) version-bumps) so the admin
evaluates provenance (§6.5 "evaluate in 30 s"; ties to `evidence_count`). **Measured against:** parent §3.2,
Slice-1 G7(7b), `SPEC.md:346,348`.

### S2-P4 — G4 surfaced, NOT baked (web-search-for-generation is a new use)
`app/domain/web_search.py` is **explicitly AC-D21-scoped to safety-link curation** (`web_search.py:1-3,157`
— "initial AC-D21 curation"). Using it for **generation** grounding is a NEW use of an AC-D21-scoped
surface — ratification-class (**G4**, §7(ii)). A2 must **surface G4** (Drive RAG and/or web-search for
grounding; RAG may be empty/topic-irrelevant) and **not bake** web-search-for-generation. **Surface =
correct, not a finding; baking web-search-for-generation = finding.** **Measured against:** parent G4 (§7),
`web_search.py:1-3`, AC-D21.

### S2-P5 — Fail-soft grounding (empty / irrelevant RAG)
Per AC-D22 / SPEC §6.1, RAG retrieval is **fail-soft**: "Drive RAG fetch failures: generation continues
without RAG context; logged." `retrieve_for_generation` returns `[]` on any error / empty index

(`rag_context` → `(none)`), `grounding_refs` honestly empty. **Measured against:** `drive_rag.py:650`
docstring, `generation.py:23-25`, SPEC §6 error handling.

docstring). A2's grounding retrieval must preserve this cost capture (the retrieve embed is real spend).
**Measured against:** `drive_rag.py` retrieve-cost docstring, AC-CD8 v1.6.
