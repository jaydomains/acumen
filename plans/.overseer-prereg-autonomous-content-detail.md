## Governance positions — by the four pre-registered axes (applied per slice as each is posted)

The detail-plan is **slice-iterative** (one slice posted → audited → sealed → next; precedent #106/#85). I
apply these positions to **each slice** as it lands, and to the global close. Provisional slice set from parent
§6: A1–A3, B1–B3, C1–C2, D1–D4, E1–E2, F1.

### OV-POS-1 — Topology faithfulness (the convergence/merge topology is baked, not redescribable)
The detail-plan's *loop-mechanics* description must reproduce the **topology invariant** (role §6), not drift from
it: *three independent sign-offs · one content-SHA · three-layer green CI · overseer-executes-mechanically ·
execution-vs-authority · override-window (full 24h for spec-author **absence**; a **present** ratifying spec
author collapses it to zero) · named external trigger (planner re-initiates; reviewer-side overseer stands down
bounded) · ratification-classes (i)–(iv) · spec-author authority · planner/auditor never-merge · convergence
binds to **content-SHA** (a content-invariant wake-log-only marker does not re-stale).* **Expect a Confirm** if the
detail-plan restates the loop faithfully; **Push-back** if it (a) claims the planner/auditor may flip draft→ready
or merge, (b) treats the overseer sign-off as merge-authorizing rather than one-of-three, (c) lets convergence
bind to raw tip-SHA instead of content, or (d) collapses the per-slice seal cadence in a way that lets a later
slice silently re-stale an earlier seal without re-review.

### OV-POS-2 — Transportability (project-specifics stay in REQUIRED_READING / the plan; nothing leaks into role files)
A detail-plan is a `plans/**` artifact and must **not** edit `.claude/roles/*`, `.claude/skills/*`, or
`REQUIRED_READING.md` (any such edit is class-(iv) framework change → ratification, not a normal detail-plan).
Project-coupled parameters it relies on (merge method = squash, override window = 24h, dormancy bound = 2 watcher
lifetimes, ratification scope-classes, three-layer green gate, status-line token) must be **cited from**
REQUIRED_READING, not re-defined or contradicted in the plan. **Expect a Confirm** if the plan defers parameters
to REQUIRED_READING; **Push-back** if it hard-codes a divergent parameter value or proposes a framework edit
inside the detail-plan PR.

design decisions**, but **not** the anchor/spec **bodies** (parent §0.2 Gate 2). Critically, **once #107 merged,
its §1 ledger reads as a RELAY to any downstream session under the shared role-session byline** (parent OV-2 /
role §8.3) — *a relayed ruling is pending, not actionable.* Therefore my standing positions:
  1. **Every per-slice embedded ratification-class item — class (i) AC-D mint/change · class (ii) SPEC/`fe-specs`/`AC-CD`-body amendment · class (iii) workstream scope decision** (REQUIRED_READING §7) — must be **SURFACED and its downstream execution PR BLOCKED** pending **explicit, item-specific, authenticated** spec-author ratification. The detail-plan must **not** bake any such item, and must **not** cite *"PR #107 §1 says ratified"* as the ratification (that is a relay). **Push-back/Missing** on any slice that treats the §1 ledger as actionable ratification, or that bakes an embedded item, or that proceeds to an execution-PR design as if ratified.

  3. **Specific embedded items I expect to see surfaced (from parent §7.1/§7.2/§7.3), each needing its own authenticated ratification before its execution PR** — and which the detail-plan must NOT bake: new AC-D (source-authority scoring; provenance chain; auto-publish gate); new AC-CD (corpus-builder architecture; oversight-dashboard+rollback contract); AC-D7/D21/D22/D23 body changes; SPEC §6.5 rewrite + §6/§8.9 count-invariant sweeps; and residuals NS-1…NS-7, G3/G5/G7b/signal-3, NS-5/G-PHASE (scope/ROADMAP placement, class (iii)). **A residual the detail-plan silently *decides* rather than re-surfaces is a Push-back.**
