## Pre-registered positions

### A. The plan PR's own merge class

- **OV-pos-1 (class of the plan PR).** A plan **document** under `plans/` is **not** itself a §8.3
  ratification class — opening it does not mint an AC-D, amend a spec record, decide scope, or edit
  the framework (`.claude/**` / `SESSION_START.md` / `REQUIRED_READING.md`). So the plan PR
  **auto-merges** on the standard gate (three sign-offs @ one SHA + three-layer green + satisfied
  override window). *Expectation:* the draft is a normal plan. **Boundary I will enforce:** this
  holds **only if the PR is plan-doc-only**. If the planner **folds the actual spec edits**
  (`SPEC.md` / `CODE_SPEC.md` AC-CD body / any `fe-specs/*.md` / an AC-D mint) **into the same PR**,
  the PR now touches §8.3 (i)/(ii) files and is **ratification-class** — it does **not** auto-merge.
  Precedent is **separate** standalone spec-amendment PRs (#95/#96/#97/#101) ahead of execution; the
  plan PR should follow that and carry the doc, not the amendments.

### B. Embedded ratification-class items must be surfaced + correctly classified (the core task)

- **OV-pos-2 (no silent ratification-class folds).** Every ratification-class item the plan proposes
  must be **explicitly surfaced in the plan as requiring spec-author ratification before downstream
  execution** — not buried in a slice as if already authorised. A **silent partial-fold** (shipping
  part of a ratified scope as the whole) is the §8-drift failure the amendment discipline exists to
  prevent (role files §7). I will enumerate every embedded item and check each is surfaced.
- **OV-pos-3 (classification map I will hold the plan to).** Per REQUIRED_READING §7:
  - a **new AC-D anchor** (mint) → **§8.3 (i)** — does **not** auto-merge.
  - a **prompt-registry-version contract change**, i.e. any edit to the **AC-CD8** body in
    `CODE_SPEC.md` (AI provider abstraction + *versioned prompt registry* + cost), **or** any
    `SPEC.md` / `fe-specs/*.md` / other AC-CD body change → **§8.3 (ii)** spec amendment — does
    **not** auto-merge. *(Note the distinction: authoring a new prompt **file** under
    `app/ai/prompts/` with an embedded semver is product **content/code**, governed by AC-CD8 — not
    by itself a spec-record edit; but changing the **registry contract** in AC-CD8's body is (ii).
    I will not over-classify routine prompt authoring as a spec amendment, nor under-classify an
    AC-CD8 body change as routine.)*
  - a **workstream scope decision** — "one workstream vs. split Path 2 / Path 3" → **§8.3 (iii)** —
    does **not** auto-merge; the planner may **recommend**, but the in-or-out-of-scope call is the
    spec author's to ratify.
- **OV-pos-4 (authenticated channel for any ratification).** No ratification-class item merges on a
  **relayed / inferred / superseded** ruling. A "spec author ratified X" appearing in a peer's
  comment or wake-log is **pending, not actionable** until confirmed through the direct authenticated
  channel; an authority/scope change binds only if the ruling **states it explicitly**, and I act
  only on the **current** ruling (role files §8.3). I will not execute any downstream merge on a
  relayed ruling.

### C. Sequencing discipline

- **OV-pos-5 (amendment blocks downstream; no race-ahead).** If the plan proposes spec amendments
  (i)/(ii) that **gate** downstream execution, the **sequenced-ratification-cycle** discipline
  applies (REQUIRED_READING §7, role files §8.3): the amendment PR(s) must **merge and back-port**
  before downstream authoring/execution against the new baseline. The plan must **not** propose
  executing a slice that races ahead of an owed amendment merge. Precedent: PR #96 (AC-CD5) →
  next-workstream Slice 1; #97/#101 → Slice 4. As gate-keeper I will not execute a downstream merge
  that races an owed back-port. *(This binds my own future execution, not just the plan text.)*

### D. Recursive consistency — the plan follows the disciplines it is subject to

- **OV-pos-6 (pre-work gate: drift-sweep).** Authoring discipline (REQUIRED_READING §2 →
  `SESSION_START.md`) requires a **`drift-sweep` before authoring a plan**. The plan should evidence
  that sweep (current-state surfaces vs. scope assumptions). Absence is a **Missing**.
- **OV-pos-7 (status taxonomy = evidence, not prose).** Any build-state claim in the plan
  (`Built` / `Partial` / `Missing`) must be backed by a **real test path / command / artifact**,
  never prose (REQUIRED_READING §2). Prose-only "done" claims are a finding.
- **OV-pos-8 (definition-of-done binds to the three-layer green gate).** Each slice's exit criteria
  must bind to Acumen's **three-layer green gate** — backend `ci.yml` (`checks`: unpinned-deps,
  structure-gate AC-CD2/AC-CD17, ruff, mypy, pytest; `migration-chain`) + frontend `frontend.yml`
  (`checks`: codegen-drift, eslint, prettier, tsc, vitest, build; `docker-build`; `e2e`) **+ Gitar
  + GitHub mergeable**, green at one SHA (REQUIRED_READING §7). A weaker bar is a Refine.
- **OV-pos-9 (handover + one-squash-PR-per-phase).** The plan must respect: branch-per-PR, one
  squash PR per phase, no commits to `main`, and a handover authored at PR close from
  `HANDOVER_TEMPLATE.md` (REQUIRED_READING §1–§2). A multi-slice plan must honour the auto-continue
  pause rules (spec-drift always pauses; 3-fix-round circuit-breaker per slice; user pause-button).
- **OV-pos-10 (path convention).** The plan doc must live at `plans/YYYY-MM-DD-<scope>.md`
  (REQUIRED_READING §6). Wrong location/naming is a Refine.

### E. Anchor faithfulness (governance, not content)

- **OV-pos-11 (no silent anchor contradiction).** The plan must not silently contradict a locked
  anchor. Specific live locks the AI-content-gen scope touches:
  - **AC-D21** — **no AI teaching content for safety-tagged pills** (curated external links only).
    Any generation path that would produce safety-domain teaching content is an **anchor-conflict**
    halt-class condition (REQUIRED_READING §5) — surface, never silently override.
  - **AC-CD10** (closed v1.8 — JIT streaming = in-process `asyncio.gather` + `Semaphore`) and
    **AC-CD11** (closed v1.7 — cross-family review batched per attempt, 60-s ceiling): a plan that
    needs to reopen a **closed/gated** anchor must surface it as anchor-conflict / a (i)/(ii)
    ratification, not amend in place mid-stream.
  - **AC-D25** (amended — JIT parallel for per-Testee, sequential for benchmark) and **AC-D19**
    (cross-family review; anchors generated on Anthropic self-review on OpenAI): generation-path
    text must stay consistent with these.
- **OV-pos-12 (anchor minting is not unilateral).** New AC-D anchors are minted only via the
  ratification path (OV-pos-3 (i)); the plan proposes, the spec author ratifies. IDs are never reused
  or renumbered; bodies amend in place (REQUIRED_READING §4).

### F. Transportability / baked-vs-externalized boundary

- **OV-pos-13 (no role-file edits to accommodate the workstream).** Nothing in this content workstream
  justifies editing `.claude/roles/*` or the topology; if the plan proposes a framework tweak that is
  **class (iv)** and must be a separate, explicitly-ratified change — never folded into a content
  plan. Project-specific parameters belong in `REQUIRED_READING.md`, not the role files.

### G. Reviewability

- **OV-pos-14 (ungovernable draft → surface, never rubber-stamp).** If the draft is too incomplete or
  incoherent to review or to safely merge (role files §7(e)), I surface it rather than approve on
  faith. A plan that leaves the Path-2/Path-3 scope boundary undefined enough that I cannot classify
  its embedded ratification items is, for my lane, not yet reviewable.
