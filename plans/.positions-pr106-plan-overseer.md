## B. Pre-registered governance positions — Slice 1

Stable, never-reused IDs, per-slice namespaced (`OV-S1.*`). Each seeds the finding set.

- **OV-S1.1 — PR-class correctness (merge-executor lane).** The detail-plan PR must remain a
  **NORMAL merge-class** PR: `plans/**`-only, baking **no** ratification-class change. Detailing
  Slice 1 to build-depth must **not** tip into *baking* a §7 decision (e.g. pre-deciding G1
  extend-vs-mint, or G7 versioning) — surfacing must stay surfacing. Any `app/`, `frontend/`,
  `SPEC.md`, `CODE_SPEC.md`, `DECISIONS.md`, `fe-specs/*`, or `.claude/*` edit would convert the PR
  to ratification-class (§8.3 (i)/(ii)/(iv)) and I would **not** auto-merge it.

- **OV-S1.2 — Two-gate distinction preserved.** Slice 1 must reproduce the parent plan §8 two-gate
  model: **Gate 1** (this plan-doc PR auto-merges on three sign-offs at one SHA + the three-layer
  green gate + the 24h override window) is **independent** of **Gate 2** (each §7 item ratified
  separately, downstream, before its execution PR). Merging the detail plan must **execute nothing,
  amend no spec, and ratify no §7 item**.

- **OV-S1.3 — Per-item authenticated ratification dependency recorded.** Slice 1 must explicitly
  enumerate which §7 items it depends on (parent plan maps A1 → **G1, G7**; plus workstream-level
  **G-SEQ, G-PHASE**) and record that **each** needs its **own explicit, authenticated, current**
  spec-author ratification through the **direct authenticated channel** *before* Slice 1's
  downstream amendment/execution PR proceeds. **No blanket, relayed, or inferred ratification** —
  the PR #104/#105 near-miss precedent stands (a relayed ruling carries the same shared byline a
  genuine one would, so it is pending, not actionable).

- **OV-S1.4 — Conditionality preserved (detailing ≠ un-gating).** The parent plan §6 marks the
  slice breakdown **"NOT authorised until §7 is ruled."** Detailing Slice 1 to build-depth must
  **preserve that conditionality** — it must not present Slice 1 as ready-to-cut. No code lands
  until a **fresh** session implements against the spec-clarification PR(s) the rulings produce
  (`SESSION_START.md` — spec drift is surfaced, never silently resolved; the implementing session
  does not author the clarification).

- **OV-S1.5 — Slice independence / dependency recording.** Slice 1 must honestly record (a) its
  place in the verified Stage A→B→C→D dependency chain (Path 3 is **structurally blocked** on
  Path 2 — verified at `catalogue.py:488` / `pill_proposal.py:27–28`, not asserted), and (b) any
  intra-Stage-A prior-slice dependency. If Slice 1 is a foundation other slices call, that is
  recorded; if it is shippable standalone (admin-driven generation), that is recorded — neither
  silently assumed.

- **OV-S1.6 — Transportability of decisions.** Decisions, rationale, and **file:line current-state
  evidence** must be recorded so they **survive into the downstream execution PR a fresh session
  picks up** — the detail plan *is* the transport. A buried ratification gate or an unrecorded
  dependency the downstream executor would lose is a transportability failure (role files §2 —
  transportability is a load-bearing governance check).

- **OV-S1.7 — Per-slice sealing topology (not conflated with merge convergence).** The
  slice-iterative **per-slice seal/final-marker** must **not** be conflated with the **global**
  convergence marker that authorizes merge. Per-slice seals are SHA/content-bound *intermediate*
  checkpoints; a later edit to a sealed slice **re-stales that seal**. The merge gate remains the
  **global** three independent sign-offs at one content-SHA + green + override window (§8 topology,
  **non-tunable**). A per-slice seal **never** authorizes merge; convergence binds to the
  **content-SHA**, and a content-invariant marker does not re-stale.

- **OV-S1.8 — Newly-surfaced ratification-class items inherit the discipline.** If detailing Slice 1
  surfaces a **new** ratification-class item beyond the 12 (a new AC-D / AC-CD anchor, a new
  `fe-specs/*` section, a class-(iii) sub-scope decision), it inherits the **same per-item
  authenticated-ratification** discipline and must be **flagged as such — surfaced, not baked**.

- **OV-S1.9 — Recursive consistency / loop-mechanics faithfulness.** Slice 1's described loop
  mechanics must match the framework producing it: **reviewer markers on reviewer branches**
  (off-canonical, content-bound); **producer marker content-invariant** on the canonical branch;
  wake-log-per-commit discipline; watcher `SELF_EXCLUDE` anchored to the **exact** branch and
  `WATCH_INCLUDE` scoped to the **real** reviewer ref-space (the PR #105 lesson: a role-named token
  matched neither `claude/<random>` branch and blinded the watcher); and **sequenced-ratification-
  cycle** handling (spec-amendment PR → gated execution PR; precedent PR #96 → Slice 1) **across the
  chain with no between-link stand-down**.

- **OV-S1.10 — Halt-class / verify-before-write discipline.** If Slice 1's detail planning hits a
  blessed halt-class condition (`REQUIRED_READING.md` §5: spec-drift, plan-unworkable,
  anchor-conflict, five-five-deadlock, precondition-failure), it must **surface by category**, not
  bake a resolution. Build-readiness / "missing-unbuilt" claims must be re-grounded in the **live
  tree at file:line** at the detail-plan SHA, never merely inherited from the parent plan.
