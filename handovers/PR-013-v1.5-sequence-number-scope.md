## Drift flags raised and how they were resolved

- **Spec/schema gap (the reason for this PR).** P4 attempt-lifecycle
  code computed `Attempt.sequence_number` as `len(prior_attempts)+1`,
  which races under concurrent `start_attempt` (TOCTOU — two parallel
  starts both read N and both write N+1). The correct fix is a DB
  uniqueness constraint, but choosing its column tuple exposed that §5's
  "per Testee per pill" scope was incoherent for a per-`Attempt`
  (per-Test) column. Resolved by correcting the spec scope to "per
  Testee per Test" and recording the canonical constraint
  `unique (test_id, testee_id, sequence_number)` in CODE_SPEC §4 (this
  PR). Spec corrected; implementation deferred to P4 with a tracked
  landing point (see open questions).
- **DECISIONS end-of-file footer — pre-existing drift, left as-is
  (scope discipline).** The footer has read v1.2 since before v1.3 — it
  was already stale before this PR, not drift introduced here.
  Untouched per the header-only decision (PR-011 precedent). Still
  recommend a dedicated doc-hygiene pass to reconcile it; it is now
  3 versions behind, independent of this amendment.
- **CODE_SPEC end-of-file footer — left at v1.4 (PR-011 precedent,
  flagged).** PR-011 bumped the CODE_SPEC footer to v1.4 alongside its
  top line; this PR follows the *header-only* decision instead, which
  reintroduces a top(v1.5)/bottom(v1.4) intra-document contradiction.
  This is a deliberate consequence of the header-only scope, not an
  oversight — flagged for the doc-hygiene pass.
- **CODE_SPEC lines 3–4 `(functional, v1.2)` / `(product anchors …
  v1.2)` companion references — left as-is (out of scope).** These are
  provenance statements about where unchanged content originated, not
  pairing/status lines, and are unrelated to AC-D3. Same treatment as
  PR-011 gave the `system_settings` defaults-provenance line. Flagged
  for a future maintenance pass.
- **SPEC amendment tally `+ 1 v1.5`.** The SPEC status line's
  amendment count gained `+ 1 v1.5` to stay consistent with how v1.3
  and v1.4 clarifying amendments were each recorded (PR-011
  precedent). This is a tally-consistency edit, not silent scope creep
  — flagged here explicitly.

## Open questions deferred to a later phase

- **Unique-constraint migration — P4.** Adding
  `unique (test_id, testee_id, sequence_number)` on `attempt`
  (whatever the next migration number is when P4 resumes) lands **in P4
  when P4 resumes, after the consolidated spec audit completes**. P4
  was PR #10 and PR #12, both closed unmerged — P4 has not landed. v1
  has **no production data**, so there is no backfill and no risk of
  pre-existing duplicate `(test_id, testee_id, sequence_number)` rows.
- **`start_attempt` concurrency strategy — P4.** When P4 resumes,
  `start_attempt` must NOT compute `sequence_number` from a pre-read
  `len(prior_attempts)+1` counter. It must rely on the DB uniqueness
  constraint and handle the collision via `IntegrityError`-retry (or an
  equivalent atomic insert-and-retry). The whole point of this
  amendment is that the constraint is the source of truth, not an
  application-side count.
