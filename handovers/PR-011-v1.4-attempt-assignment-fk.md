## Drift flags raised and how they were resolved

- **Spec/schema gap (the reason for this PR).** AC-D26 requires
  `engagement_status` to derive per assignment, but the P1 schema
  (migration `0002`) has no Attempt → Assignment FK. Resolved by making
  the link explicit in the spec (this PR); the column itself is deferred
  to migration `0004` in P4 (see open questions). Spec corrected;
  implementation deferred with a tracked landing point.
- **CODE_SPEC pairing-line drift (pre-existing).** `CODE_SPEC.md`
  Status read "Paired with SPEC.md v1.2 / DECISIONS.md v1.2" while the
  canonical docs were already at v1.3 before this PR. Corrected to v1.4
  as part of this amendment (user-approved), which also closes the
  older v1.2→v1.3 lag.
- **CODE_SPEC `system_settings` defaults-provenance line — left as-is
  (out of scope).** Further down §4, the line "`system_settings`
  columns (defaults from `DECISIONS.md` v1.2)" still reads v1.2. That
  is a factual statement about where those (unchanged) defaults
  originated, not a pairing/status line, and is unrelated to AC-D26.
  Deliberately not touched under "clarification only — no other
  changes." Flagged for a future maintenance pass if the project wants
  provenance lines kept current.
- **CODE_SPEC bottom footer — bumped to v1.4 (consistency).** The
  end-of-file footer (`*End of Acumen CODE_SPEC … Paired with SPEC.md
  v1.2 and DECISIONS.md v1.2*`) was bumped to v1.4 alongside the
  top Status line. Leaving it at v1.2 after bumping the top would have
  introduced a new intra-document top/bottom contradiction — the user's
  explicit "SPEC.md status line (top and bottom)" instruction
  establishes that a doc's status lines are kept consistent, so the
  same was applied to the doc whose status was bumped.
- **DECISIONS bottom footer — pre-existing drift, left as-is (in
  scope-discipline).** `DECISIONS.md` line 632 footer reads "Status:
  v1.2. Paired with SPEC.md v1.2" and its amendment-count prose ("6 …
  v1.1, 3 in v1.2 plus new AC-D27") never mentioned v1.3 either — it
  was already stale before this PR, not drift introduced here. The user
  scoped DECISIONS edits explicitly to "header and amendment table"
  (deliberately asymmetric vs SPEC's "top and bottom"), so the footer
  was left untouched. Recommend a dedicated doc-hygiene pass to
  reconcile this footer (it is 2+ versions behind independent of this
  amendment).
