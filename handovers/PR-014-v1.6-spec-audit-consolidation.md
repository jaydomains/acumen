## Drift flags raised and how they were resolved

- **Schema-vs-doc verification (the core finding).** `app/models.py` +
  migration `0002_p1_data_model.py` (which compiles DDL directly from
  `Base.metadata.sorted_tables`) were read before drafting edits. Result:
  `assignment_assignee`, `assignment_reminder`, `attempt_pause_event`,
  `attempt_focus_event` **all exist in the landed P1 schema**;
  `learning_material.served_at`/`served_text` exist; `grade_review` is 1:1
  and already carries OpenAI provenance via `AIProvenanceMixin`; that mixin
  is on `grade`, `grade_review`, `question`, `anchor_question`,
  `weakness_report`, `learning_material`; `system_settings` already has
  `model_by_operation`, `provider_by_operation`, `review_provider`.
  **No P1 patch migration is required** (the brief's "bigger problem" branch
  did not trigger). v1.6 resolution: document the shipped schema.
- **Verification found drift the audit did NOT flag.** CODE_SPEC §4's
  Question row listed a column `is_anchor` that does **not** exist in
  `app/models.py` (anchor-ness is via `question.pill_id` + the separate
  `anchor_question` table). Corrected in v1.6 to the shipped three-way owner
  FK. A second ripple was caught by the post-edit grep set: the AC-CD7 §18
  anchor decision line still read "six crons" after §8 / §8.9 / ROADMAP /
  CHECKLIST were updated to seven — corrected so no doc is left at "six".
  Both confirm the verification step is doing real work — it is not
  rubber-stamping the audit; it caught spec/schema and intra-doc mismatches
  beyond the 18 audit findings.
- **F2-reminder / F3 / F11 would have required new schema.** Documenting
  per-assignee reminder rows / `system_settings.grade_review_reconcile_
  interval_minutes` / `grade_review_max_retry_attempts` as the brief stated
  would push the docs *ahead* of the shipped schema — the exact drift this
  audit exists to prevent. Resolved by behavioural-default framing (decision
  2 above); the real columns land when P6 builds them (tracked below).
- **DECISIONS / CODE_SPEC / SESSION_START stale footers.** Pre-existing,
  left as-is per PR-011/PR-013 scope discipline. Tracked below.
