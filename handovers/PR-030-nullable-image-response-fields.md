## What was decided in this PR

- **D1 — Touch both Pydantic surface AND testee-facing dict surface.** Investigation (Phase 1) revealed that `QuestionResponse` at `app/schemas.py:406` is *not* the attempt-facing payload — it's only used by admin Test-management routes (`POST/GET/PATCH /v1/tests/{test_id}/questions` at `app/routers/tests.py:179,198,214`). The attempt payload (`AttemptView.questions: list[dict] | None`, `AttemptView.q1: dict | None`) is a plain dict built in `_present_one()` at `app/domain/attempts.py:503`. The PR description literally said "find QuestionResponse" but its stated intent was the testee-facing payload. Surfaced both via `AskUserQuestion` at plan time; user confirmed "Both — admin surface and testee-facing surface", so the PR touches both. Frontend gets nullable image fields on both surfaces; no refactor when visuals ship at either entry point.
- **D2 — Change MCQ option shape from `list[str]` to `list[{text, image_url}]` with backward-compatible legacy-string tolerance.** Investigation revealed there is no `Choice` Pydantic class; MCQ options are `list[str]` validated at `app/domain/tests.py:281-298`. The PR description asked for `image_url: str | None = None` on the Choice response — impossible without changing the option shape (a string cannot carry an `image_url` key). Surfaced via `AskUserQuestion`; user confirmed the shape change is intentional and in-scope ("dropping the choice field defeats the PR's purpose"). The presentation layer (`_present_one`) always emits the dict shape; storage validation tolerates both; AI generation continues emitting `list[str]`. When v1.x visuals ship, the legacy-string tolerance can be removed in that PR.
- **D3 — `ChoiceResponse` Pydantic class added as informational (OpenAPI-only).** The class is not directly referenced by any response model field (`QuestionResponse.config` is still `dict`). Its purpose is to give the OpenAPI surface a typed name so the frontend's TypeScript generator emits a `ChoiceResponse` interface instead of an untyped string-vs-dict ambiguity. The cost (one Pydantic class) is much smaller than the benefit (clean frontend type generation). Docstring explains the purpose explicitly. User-confirmed call at the ExitPlanMode boundary with the exact docstring text adopted verbatim.
- **D4 — No AC-D / AC-CD anchor amendment, no DB migration.** Spec scan confirmed option format is implementation-defined: SPEC.md:272 ("options and correct answer for deterministic types"), SPEC.md:304 ("options + correct key for MCQ"), and DECISIONS.md:594 (AC-D24 `randomise_option_order`) all describe semantics without pinning shape. The `list[str]` constraint lived only in implementation code (`app/domain/tests.py` validator + `app/ai/prompts/generation.py:75-79` prompt narrative). SPEC.md:417 already anticipates "image-based question media that emerges in v1.x" — this PR makes the API contract honour that intent without shipping the feature. No `Question` ORM column is added; Pydantic `from_attributes=True` + `None` defaults handle the missing-attribute case. When v1.x visuals actually ship, that PR adds the columns + a migration, and the response surface light-up automatically.
- **D5 — Tolerate both option shapes in the validator during transition.** Existing test fixtures (24 files, ~50+ sites) all use `config={"options": ["a", "b"], "correct": 0}` storage. Tightening validation to dict-only would force fixture-wide churn outside this PR's scope. The tolerant validator (accept string OR `{text, image_url}` dict) means: (a) all existing integration tests continue to pass without fixture changes (429 integration tests verified green); (b) admin POST/PATCH continues to accept both shapes; (c) future visual-content storage can land without re-validating already-stored questions.
- **D6 — Presentation always emits the dict shape regardless of input.** `_present_one()` uses `_wrap_option()` to normalise — legacy string → `{text: "...", image_url: None}`, pre-wrapped dict → pass-through preserving `image_url`. This is the wire contract the frontend codes against; storage shape is implementation-detail upstream. The frontend never sees a string option from this PR onward.
- **D7 — AI generation + stub kept as-is.** `app/ai/provider.py:90` (stub) and `app/ai/prompts/generation.py:75` (real-provider prompt) continue emitting `list[str]`. The presentation-layer wrap is the bridge. When v1.x visuals ship, the generation prompt + stub flip to emit dicts directly, and the legacy-string tolerance in `_present_one()` + `_valid_option()` can be removed in that PR. Keeps this PR's scope strictly response-shape forward-compat, not generation-shape change.
- **D8 — Handover authored for this PR.** Confirmed at the original prompt ("Authored handover"). Matches the per-PR handover discipline. Sections-strict per SESSION_START.
- New anchors introduced: **none**.
- Existing anchors referenced (no amendments): SPEC.md:272/304 (Question entity + generation outputs — option format not pinned), SPEC.md:417 (anticipates "image-based question media that emerges in v1.x"), AC-D24 (`randomise_option_order` semantics unchanged), AC-D5 (Question entity unchanged at DB level), AC-CD2 (router/domain layout untouched), AC-CD15 (zero-network test discipline — new unit test is pure schema + presentation assertions, no DB / no network).

## Anything a fresh Claude Code session needs to pick up cleanly

- **`QuestionResponse` is the admin Test-management Pydantic class, not the attempt-facing payload.** Easy mistake — the name suggests "question response" generically. The attempt-facing question payload (testee-facing) is a plain dict built in `_present_one()` at `app/domain/attempts.py:503-538`. Both surfaces now carry the v1.x visual fields; do not assume only one of them does.
- **MCQ options now have two valid shapes on storage but exactly one on the wire.** Storage (`Question.config`): `list[str]` (legacy, what AI generation emits today) OR `list[{text: str, image_url: str | None}]` (forward-compatible). Wire (`_present_one()` output): always `list[{text, image_url}]` — the normalisation happens in the presentation layer. If you add a new code path that reads `options` from `config`, decide explicitly whether it's reading storage (handle both shapes) or reading the presented view (always dict). The `_wrap_option` helper at `app/domain/attempts.py` is the single normalisation point.
- **No DB column was added for `reference_image_url` / `reference_image_caption`.** They live in `config` (JSONB) when populated. The Pydantic `QuestionResponse.from_attributes=True` + None defaults handle the absent ORM attribute case. When v1.x visuals actually ship, that PR adds real columns to `acumen.question` and the response surface picks them up automatically.
- **The PR introduces `ChoiceResponse` as an informational Pydantic class.** It exists for OpenAPI TypeScript generation — the frontend's type generator will emit a `ChoiceResponse` interface. It is NOT referenced by any backend response field today (because `QuestionResponse.config` is still `dict`). If a future change wants to type the embedded options on the wire, `QuestionResponse.config` becomes a typed model with a `list[ChoiceResponse]` field for MCQ — but that's a much larger refactor (would touch every test fixture) and is out of scope here.
- **AI generation + admin authoring still emit `list[str]`.** Deliberate (D7). The presentation-layer wrap is the bridge. When v1.x visuals ship: (a) flip `app/ai/prompts/generation.py:75-79` prompt to ask for `{text, image_url}` objects; (b) flip `app/ai/provider.py:90` stub to emit objects; (c) optionally tighten `_valid_option` to dict-only.
