## What was decided in this PR

- New anchors introduced: **none.** This amends existing AC-D25
  Implications and clarifies the existing AC-CD10 body — both keep
  their identifiers.
- Existing anchors this PR depends on: AC-D25 (product, JIT
  streaming), AC-CD10 (technical, §10 streaming buffer mechanics
  & failure handling), AC-D11 (product, pause mechanism — reused
  for the single-failure disposition), AC-D17 (product, frozen-
  test snapshot — reused for resume semantics), AC-D24 (product,
  shuffle seed — already named in §10's resume clause), AC-D22
  (product, RAG context — referenced for the shared-context
  reuse across per-question calls), AC-CD15 (technical, zero-
  network test harness — informed the closure direction),
  AC-CD4 (technical, entity → table mapping — extended with the
  `attempt_position` column row), and indirectly AC-D5 / AC-D20
  (product anchors that AC-D25 already cites).
- Decisions recorded with the user (plan-mode approval; three
  bound questions in one AskUserQuestion call):
  1. **Execution model: in-process `asyncio.gather` +
     `asyncio.Semaphore` (D1).** The runner-up — Celery tasks +
     Redis pub/sub — was rejected: at v1 load shape (≤30
     concurrent attempts, 5–10 Q-N tasks per attempt) the
     in-process I/O budget is comfortable; the JIT goal is sub-
     3-s Q1 + minimal between-question wait, so Celery enqueue
     + Redis result-backend round-trip is overhead in exactly
     the wrong place; AC-CD15 zero-network test harness is
     simpler in-process (no Redis pub/sub mock); Celery
     infrastructure remains scoped to its actual job (P11 cron
     set + long-running batch). Horizontal worker-death-mid-
     attempt concern is mitigated by the DB-backed
     `attempt_position` + `Last-Event-ID` resume semantics
     built into the same P10 PR — completed Question rows
     replay from any worker; nothing depends on the original
     worker surviving. The third option ("Surface as spec
     drift") was the chosen *process* (this PR), with
     asyncio.gather as the chosen *resolution direction*.
  2. **Ordering column: add `question.attempt_position` (D2).**
     Concurrent Q-N completions under `asyncio.gather` can
     stamp identical `created_at` timestamps (sub-millisecond
     resolution still collides under load); the fragile
     ordering would break snapshot-replay determinism. The
     additive nullable INT column with unique
     `(attempt_id, attempt_position)` matches P4's
     `(test_id, sequence_number)` shape verbatim. Slots are
     assigned at enqueue time (not generation-completion time),
     so the ordering is set before any task resolves.
  3. **Single-failure policy: one orchestration-layer retry,
     then AC-D11 pause (D3).** The runner-up "pause immediately
     on first failure" is simpler but turns transient blips
     into Testee-visible pauses; the orchestration-layer single
     retry sits above tenacity's HTTP-level retries inside
     `_invoke` and handles the application-layer transient
     class. The other runner-up "mark slot failed and continue
     streaming the rest" — present in the prompt as an
     alternative — was rejected: per-question gaps in an
     attempt are not a v1 shape; the snapshot-replay semantics
     AC-D17 / AC-D24 / AC-D25 share require gap-free ordering;
     and the §10 verb "retry / abandon" is Testee-facing UI
     state on the paused attempt, not orchestration-layer
     "continue past gaps". Pause via the existing AC-D11
     mechanism is the v1 disposition.
  4. **Wide file scope (PR-017 pattern).** SPEC + DECISIONS +
     CODE_SPEC + SESSION_START + ROADMAP + CHECKLIST + handover
     = 7 files, deliberately matching PR-017's "no doc left
     behind" approach to cross-doc consistency.
