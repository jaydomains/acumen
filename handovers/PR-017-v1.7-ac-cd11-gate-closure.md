## What was decided in this PR

- New anchors introduced: **none.** This amends existing AC-D19 and
  closes the existing AC-CD11 — both keep their identifiers.
- Existing anchors this PR depends on: AC-D19 (product), AC-CD11
  (technical), and indirectly AC-D2 / AC-D5 / AC-D12 / AC-D18 / AC-D23
  / AC-CD8 (the AC-D19 spec-references graph), plus the v1.6 AC-D19
  amendments (grade_review table, reconcile cron, stuck-pending auto-
  flag, mixed-test display gate).
- Decisions recorded with the user (plan-mode approval):
  1. **Batched per attempt (not parallel, not sequential).** One
     OpenAI `review()` call per submit, payload = the attempt's full
     AI-graded response list, structured response = array of
     `{grade_id, verdict, reasoning?}`. The runner-up — per-response
     parallel (N concurrent calls) — was rejected: multiplies upstream
     rate-limit load by N, complicates partial-failure UX, and buys
     no meaningful headroom at the typical 3–5-response attempt size.
     Per-response sequential was the anti-pattern that motivated F10
     in the first place (N×ceiling); not defensible.
  2. **60-second hard ceiling.** The runner-up 30 s alternative is
     tighter and matches AC-D19's upper bound exactly, but on busy
     OpenAI days it would force routine "review pending" → drives the
     fail-soft path from rare-failure to normal-path and weakens the
     AC-D19 v1.1 "no internalise-then-walk-back" guarantee. The other
     runner-up 90 s alternative trades nothing for the longer
     "checking your answers..." UX. 60 s covers the modelled worst
     case and stays squarely in the rare-failure-path regime.
  3. **F11 named, not changed.** The cron 5 min × max-retry 10 → ≈50-
     min stuck-pending auto-flag window was already locked in v1.6;
     v1.7 just names the wall-clock so the operator-visible SLA reads
     from one place (AC-D19 Implications (v1.7), AC-CD11 Decision /
     Implications, SPEC §8.9 (v1.7)). No `system_settings` columns
     added; defaults still behavioural per the v1.6 precedent.
  4. **Wide file scope (PR-014 pattern).** SPEC + DECISIONS + CODE_SPEC
     + SESSION_START + ROADMAP + CHECKLIST + handover = 7 files,
     deliberately matching PR-014's "no doc left behind" approach to
     cross-doc consistency.

  submit-path ceiling** to the same list. All three remain "P6

- **P6 must emit observability on every `review()` call.** Per the
  Risks clause in ROADMAP P6 (v1.7), the 60-s ceiling being "a single
  tunable" is rhetorical unless the future tuning pass has data to
  act on. **Mandatory P6 deliverable:** every call records at minimum
  `latency_ms`, `success` (bool), `batched_payload_size` (count of
  AI-graded responses in the batch), `ceiling_breached` (bool). The
  data lands wherever P6's review provenance lands (likely the
  existing `AIProvenanceMixin` columns on `grade_review`, plus
  whatever per-call audit log P6 settles on).
- **Preliminary-result UX (P6 frontend).** Within the 60-s window the
  Testee sees durable confirmed/flagged grades. Over the ceiling the
  result page renders in preliminary mode — and AC-D19 v1.1's "no
  internalise-then-walk-back" guarantee depends on this being
  **visibly preliminary**, not silently so. A small "review pending"
  tag is insufficient; the page must clearly communicate to the
  Testee that the result is provisional and may change after admin
  review (especially since the auto-flag window is ≈50 min). This is
  P6 frontend territory (likely out of P6 backend scope) — recording
  here so the v1.1 durability guarantee doesn't silently re-break
  when the frontend session lands.
